# AI-Human協働プロジェクト管理フレームワーク

このドキュメントは、AI（Claude）と人間が協働するプロジェクトにおいて、効率的かつ品質を担保した進め方を定義する。

## 2. バリューチェーン設計指針

### バリューチェーンとは
顧客（この場合は読者・広告主）に価値を提供するまでの一連の活動を構造化したもの。

### 設計ステップ
1. **主要活動の特定**: 価値創出に直接関わる活動
2. **支援活動の特定**: 主要活動を支える活動
3. **各活動のインプット・アウトプット定義**
4. **AI/Human役割分担の明確化**

### 本プロジェクトのバリューチェーン
詳細は [VALUE_CHAIN.md](./VALUE_CHAIN.md) を参照

## 3. タスク管理の原則

### タスク分解の粒度
- **1タスク = 1-4時間で完了できる単位**
- **依存関係が明確**
- **受け入れ基準が定義可能**

### GitHub Issueの活用
各タスクはGitHub Issueとして管理し、以下の情報を含める：

```markdown
## タスク概要
[タスクの目的と内容]

## 担当
- AI / Human / 協働

## 受け入れ基準
- [ ] 基準1
- [ ] 基準2
- [ ] 基準3

## 依存関係
- #issue_number （このIssueが完了していること）

## 成果物
- ファイルパス or URL

## 備考
[補足情報]
```

### GitHub Projectsの活用
- **ステータス管理**: Backlog / Todo / In Progress / Review / Done
- **優先度**: P0 (最優先) / P1 (高) / P2 (中) / P3 (低)
- **マイルストーン**: Phase単位で管理

## 4. 品質管理基準

### 品質チェックの二段階方式

#### 一次チェック（AI担当）
- [ ] 要件を満たしているか
- [ ] ドキュメントに誤字脱字がないか
- [ ] コードに明らかなバグがないか
- [ ] 受け入れ基準を満たしているか

#### 最終承認（Human担当）
- [ ] 方向性が正しいか
- [ ] 品質基準を満たしているか
- [ ] ビジネス目標に合致しているか
- [ ] 公開・コミットして問題ないか

### ドキュメントの品質基準
- **明確性**: 目的が明確で、読み手が理解できる
- **完全性**: 必要な情報が全て含まれている
- **一貫性**: プロジェクト内の他ドキュメントと矛盾がない
- **保守性**: 更新・変更が容易

### コードの品質基準
- **動作性**: 意図した通りに動作する
- **可読性**: コメント・命名規則が適切
- **保守性**: 後から修正しやすい構造
- **セキュリティ**: 脆弱性がない

### コンテンツ（ブログ記事）の品質基準
- **価値提供**: 読者にとって有益な情報
- **SEO最適化**: キーワード・メタデータが適切
- **オリジナリティ**: コピーコンテンツでない
- **正確性**: 技術的に正しい内容
- **読みやすさ**: 構成・文章が分かりやすい

## 5. リスク・課題管理

### リスク管理の基本方針
1. **予防**: 事前にリスクを洗い出し、対策を講じる
2. **早期検知**: 問題が小さいうちに発見する
3. **迅速対応**: ブロッカーが発生したら即座に対処

### リスクカテゴリー
- **技術リスク**: 技術的な実現可能性、学習コスト
- **収益リスク**: マネタイズの不確実性
- **時間リスク**: スケジュール遅延
- **品質リスク**: 成果物の品質低下

### 課題管理フロー
```
発見 → Issue化 → 優先度設定 → 対応策検討 → 実施 → クローズ
```

### GitHub Labelの活用
- `risk`: リスク項目
- `bug`: バグ・不具合
- `blocker`: 作業がブロックされている
- `question`: 不明点・質問
- `enhancement`: 改善提案

## 6. AI-Human協働の実行サイクル

### 基本サイクル
```
1. AI: タスク内容の確認・質問
   ↓
2. Human: 指示・承認
   ↓
3. AI: 作業実施・一次チェック
   ↓
4. AI: 成果物提示・レビュー依頼
   ↓
5. Human: 最終レビュー・フィードバック
   ↓
6. AI: 修正（必要に応じて）
   ↓
7. Human: 承認・マージ・公開
   ↓
8. 両者: タスク完了確認・次タスクへ
```

### コミュニケーションルール

#### AIの原則
- **提案ベース**: 「〜しましょうか？」「〜でよろしいですか？」
- **不明点は質問**: 推測で進めない
- **進捗報告**: 長時間かかる作業は途中経過を報告
- **選択肢提示**: 複数案がある場合は比較表を提示

#### Humanの原則
- **明確な指示**: 曖昧な指示は避ける
- **フィードバック**: 良い点・改善点を具体的に
- **優先度明示**: 「これは重要」「これは後回し」を伝える

## 7. 進捗管理・報告

### マイルストーン管理

- **各Phaseの目標を明確化**
- **Phase完了時に振り返り**

## 8. ドキュメント更新ルール

### ドキュメント更新の自動チェック機構

#### GitHub Actions による週次自動チェック
毎週日曜日 9:00 JST に、GitHub Actions が自動的に「週次ドキュメント更新チェック」Issueを作成します。

**ワークフローファイル**: `.github/workflows/weekly-doc-check.yml`

**チェック内容**:
- CLAUDE.md の更新必要性確認
- PROJECT_FRAMEWORK.md の更新必要性確認
- その他プロジェクトドキュメントの鮮度確認

**手動実行**: GitHub Actions の画面から手動でも実行可能

#### 更新トリガー

##### 1. 自動トリガー（GitHub Actions）
- **週次チェック**: 毎週日曜日に自動Issue作成
- **Phase完了時**: Phase完了時に手動でワークフローを実行推奨

##### 2. 即時更新が必要なケース
以下の場合は、週次チェックを待たずに即座に更新：

**CLAUDE.md**:
- [ ] 重要な意思決定があった
- [ ] プロジェクトの目標・方向性が変わった
- [ ] フェーズが進んだ
- [ ] 役割分担が変わった
- [ ] 新しい制約・ルールが追加された

**PROJECT_FRAMEWORK.md**:
- [ ] プロジェクト管理手法に問題が見つかった
- [ ] 新しいベストプラクティスが発見された
- [ ] 品質基準を見直す必要が生じた
- [ ] タスク管理方法を改善したい

#### 更新プロセス

```
┌─────────────────────────────────┐
│ トリガー発生                      │
│ (週次自動 / Phase完了 / 即時)    │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│ GitHub Actionsが                 │
│ チェックIssueを自動作成           │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│ AI: Issueのチェックリスト確認     │
│ - 更新必要性を判断               │
│ - 変更箇所を特定                 │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│ AI: Issue にコメント              │
│ 「以下の更新が必要です」          │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│ Human: 承認 or 却下               │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│ AI: PRで更新ドラフト作成          │
│ Human: レビュー・承認             │
│ マージ → Issue クローズ           │
└─────────────────────────────────┘
```

### CLAUDE.mdの更新
- **タイミング**: 上記トリガーに従う
- **担当**: AIがドラフト作成 → Humanが承認

### プロジェクトドキュメントの更新
- **原則**: ドキュメントは常に最新状態を保つ
- **方法**: PRベースで更新し、レビューを経てマージ

### 更新履歴の記録
- **Gitコミットメッセージで記録**
- **重要な変更はCHANGELOG.md（必要に応じて作成）**
